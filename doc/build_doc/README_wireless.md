# Wireless 模块移植说明

本文档面向“把 `App/Wireless` 移植到其他 MCU / 其他工程”的开发者，不按当前 STC 工程视角描述。阅读目标是：先弄清楚无线模块依赖哪些硬件、哪些代码必须改、哪些协议不能随便改。

> 当前实现不是通用无线库，而是“STC8G + 两颗 LT8920 + 项目业务协议”的实现。换 MCU 时不要只改几个 GPIO 宏，需要先拆清楚硬件适配层、LT8920 驱动层、业务协议层。

## 1. 移植前必须先确认

移植前先回答下面几个问题，否则后面很容易调不通：

1. 新板子是 **两颗 LT8920** 还是 **一颗 LT8920**？
2. SPI 使用 **GPIO 模拟 SPI** 还是 MCU 硬件 SPI？
3. 是否保留当前 `10ms` 周期调度？
4. 是否要兼容现有遥控器 / 对端设备的空口协议？
5. 配对信道、工作信道、同步字、包格式是否保持不变？
6. 当前业务层是否仍然直接控制电机、LED、自动驾驶等模块？

如果只是换 MCU、仍然兼容原对端，建议优先保持 LT8920 寄存器配置、信道算法、同步字和业务帧格式不变，只替换底层 GPIO/SPI/延时/日志等平台接口。

## 2. 当前代码分层

### 2.1 MCU / 板级适配层

当前没有单独抽象成 `bsp` 文件，板级依赖直接写在头文件和业务代码里。

主要位置：

- `App/Wireless/LT8920/LT8920_SPI.h`
- `App/Wireless/LT8920/LT8920_SPI.c`
- `App/Wireless/wirelessProtocal.h`
- `App/Wireless/wirelessProtocal.c`
- `App/Main/main.c`

换 MCU 时至少要提供：

| 能力 | 当前实现 | 移植要求 |
| --- | --- | --- |
| GPIO 输出 | `Pxx = 0/1` | 替换成新 MCU 的 GPIO 写接口 |
| GPIO 输入 | 直接读 `Pxx` | 替换成新 MCU 的 GPIO 读接口 |
| SPI 时钟 | GPIO 模拟 `RF_CLK_H/L` | 可继续模拟，也可改硬件 SPI |
| 延时 | `delay_us()` / `delay_ms()` | 保证 LT8920 时序需要的 us/ms 延时 |
| 日志 | `printf()` | 没串口时可改为空宏或新日志接口 |
| 基础类型 | `uint8_t` / `uint16_t` / `bit` | `bit` 是 8051 风格，换平台要替换 |
| 周期调度 | 主循环约 `10ms` 调一次 | 新系统要提供等效周期任务 |

建议移植时先新建一个板级适配文件，例如：

- `wireless_port.h`
- `wireless_port.c`

把 GPIO、SPI、延时、日志全部收口到这里，避免无线驱动继续散落 MCU 专用宏。

### 2.2 LT8920 驱动层

主要文件：

- `App/Wireless/LT8920/LT8920_SPI.h`
- `App/Wireless/LT8920/LT8920_SPI.c`

职责：

- LT8920 复位
- SPI 读写寄存器
- FIFO 读写
- 初始化寄存器配置
- 设置发射 / 接收模式
- 发送一包数据
- 查询接收完成和 CRC 状态

这一层只应该关心“怎么操作 LT8920”，不应该知道遥控命令、电机、GPS、自动驾驶等业务。

### 2.3 无线业务协议层

主要文件：

- `App/Wireless/wirelessProtocal.h`
- `App/Wireless/wirelessProtocal.c`

职责：

- 配对流程
- 工作信道计算
- 同步字配置
- 应用层帧组包 / 解包
- 遥控命令解析
- GPS / 电量 / 自动驾驶状态上报
- 遥控失联超时处理

这一层目前和业务模块耦合较重。换 MCU 时如果只想复用 LT8920 收发能力，建议不要直接复用整个 `wirelessProtocal.c`，而是先把协议收发和业务动作拆开。

### 2.4 系统调度层

当前入口在 `App/Main/main.c`：

- 上电后调用 `LT89xx_INIT()` 初始化两颗 LT8920。
- 主循环中周期调用 `Radio_progress()`。
- 主循环中周期调用 `WirelessProtocal_Accelerator_OutTime_Handle()`。

当前实现依赖约 `10ms` 周期。代码里的很多超时值不是绝对毫秒，而是基于这个周期累加出来的 tick 数。移植到 RTOS 或不同主循环时，要确认周期是否仍然等效。

## 3. 当前硬件结构

### 3.1 当前是双 LT8920 方案

代码里有两个角色：

| 角色 | 含义 | 当前用途 |
| --- | --- | --- |
| `LT8920_SEND` | 发送芯片 | 发送配对包、GPS / 状态包 |
| `LT8920_REC` | 接收芯片 | 接收遥控命令 |

也就是说，当前默认板子上同时挂了两颗 LT8920，不是一颗 LT8920 在收发之间快速切换。

如果新板子只有一颗 LT8920，不能只改片选脚。至少要重新设计：

- `Radio_progress()` 的收发调度
- 配对期间的发包和收包切换
- 工作模式下 GPS 上报和遥控接收的时分复用
- `LT8920_SEND` / `LT8920_REC` 两个角色的接口
- 超时策略和丢包容忍度

单芯片方案本质上是半双工，需要明确“什么时候发、什么时候收、发多久、收多久”。

### 3.2 当前引脚映射

以 `App/Wireless/LT8920/LT8920_SPI.h` 为准：

| LT8920 信号 | 当前 STC 引脚 | 说明 |
| --- | --- | --- |
| `MOSI` | `P35` | 两颗 LT8920 共用 |
| `SCK` | `P34` | 两颗 LT8920 共用 |
| `RST` | `P37` | 两颗 LT8920 共用复位 |
| `SEND_CS` | `P33` | 发送芯片片选 |
| `SEND_MISO` | `P36` | 发送芯片 MISO |
| `REC_CS` | `P03` 或 `P02` | 由 `HW_V16` 决定 |
| `REC_MISO` | `P02` 或 `P03` | 由 `HW_V16` 决定 |
| `PKT` | `P32` | 当前主要通过读寄存器判断包状态 |

移植时建议不要把新 MCU 引脚继续直接写进 `LT8920_SPI.h`，而是做成下面这种抽象：

| 抽象接口 | 含义 |
| --- | --- |
| `WIRELESS_SPI_MOSI_WRITE(level)` | 写 MOSI |
| `WIRELESS_SPI_SCK_WRITE(level)` | 写 SCK |
| `WIRELESS_LT8920_RESET_WRITE(level)` | 写复位脚 |
| `WIRELESS_LT8920_CS_WRITE(role, level)` | 按角色写片选 |
| `WIRELESS_LT8920_MISO_READ(role)` | 按角色读 MISO |
| `WIRELESS_DELAY_US(us)` | us 延时 |
| `WIRELESS_DELAY_MS(ms)` | ms 延时 |
| `WIRELESS_LOG(...)` | 调试日志 |

这样以后再换 MCU，只需要改 `wireless_port`，不用改协议代码。

## 4. LT8920 关键配置

### 4.1 当前主要参数

| 项目 | 当前值 / 位置 | 移植说明 |
| --- | --- | --- |
| 射频芯片 | LT8920 | 对端不变时不要替换寄存器配置 |
| 发射功率 | `RF_Power = LT89xx_6dBm` | 定义在 `LT8920_SPI.h` |
| 配对信道 | `Pair_Channel = 0x7F` | 已超出常规 `2402 + channel` 频点理解，需结合实测确认 |
| 默认测试频点 | `RadioFrequency_user = 2402` | `Test_Channel = RadioFrequency_user - 2402` |
| 空口速率 | `Air_rate_62K5` | 当前只打开 62.5Kbps |
| 配对同步字 | `SyncPairWord = E4 E4 E0 E0` | 配对阶段使用 |
| CRC | 打开 | 寄存器配置里已启用 |
| 包长度 | FIFO 包里带长度 | 业务包仍有自己的长度字段 |

如果要兼容原遥控器 / 原接收端，这些值不要随意改。改任意一个都可能导致完全收不到包。

### 4.2 `RF_Encrypt_Config()` 的真实含义

`RF_Encrypt_Config()` 这个名字容易误解。当前它主要做的是 LT8920 Sync Word 配置，不是严格意义上的数据加密。

移植文档和新代码里建议改名或至少注释清楚，例如：

- `RF_SyncWord_Config()`
- `LT8920_SetSyncWord()`

## 5. 空口协议和业务帧

### 5.1 LT8920 FIFO 包

LT8920 负责空口收发，FIFO 数据里包含长度和 payload。当前驱动会读写 FIFO，并用寄存器状态判断收包完成和 CRC 错误。

移植时先验证 LT8920 层能稳定收发固定字节，不要一开始就调业务协议。

推荐顺序：

1. 复位脚有效。
2. SPI 寄存器写入后可以正确回读。
3. FIFO 能发固定测试包。
4. 另一端能收到固定测试包。
5. 再接入 `wirelessProtocal.c` 的业务帧解析。

### 5.2 应用层业务帧

当前应用层帧以 `0xAA` 开始、以 `0xBB` 结束，中间包含长度、命令字和 payload。具体格式以 `wirelessProtocal.c` 的组包 / 解包代码为准。

目前能看到的命令包括：

| 命令 | 当前含义 | 方向 |
| --- | --- | --- |
| `0x11` | 遥控 / 加速度控制类命令 | 对端到船端 |
| `0x12` | GPS / 船端状态上报 | 船端到对端 |

如果对端固件不改，命令字、字段顺序、字段长度、大小端都必须保持一致。

## 6. 移植步骤建议

### 6.1 第一步：拆出平台接口

先处理所有 MCU 强相关内容：

- `Pxx` 端口操作
- `bit` 类型
- `delay_us()` / `delay_ms()`
- `printf()`
- 中断 / 主循环 / RTOS 定时任务
- STC 专用头文件 `STC8G_H_GPIO.h`

目标是让 LT8920 驱动层不再直接包含 STC 头文件。

### 6.2 第二步：验证 LT8920 驱动

只测芯片，不测业务：

- 初始化返回正常。
- 关键寄存器能写入和回读。
- 两个角色的 CS / MISO 没有接反。
- `LT8920_SEND` 能发包。
- `LT8920_REC` 能收包。
- CRC 错误能被识别。

双芯片板尤其要确认发送芯片和接收芯片的片选、MISO 没有互换。

### 6.3 第三步：接入配对流程

验证内容：

- 配对包能按 `Pair_Channel` 发出。
- 对端能回复配对响应。
- 能派生出工作信道 `RF_Channel[0..2]`。
- 能派生出同步字 / key。
- 两颗 LT8920 都重新配置到工作参数。

### 6.4 第四步：接入业务命令

最后再验证业务：

- 能稳定收到 `0x11` 控制命令。
- 遥控失联超时能触发。
- 能周期发送 `0x12` GPS / 状态包。
- 电机、LED、自动驾驶等业务动作符合预期。

## 7. 单 LT8920 移植特别说明

如果新硬件只有一颗 LT8920，需要把当前双芯片并行模型改成单芯片半双工模型。

建议新增一个无线状态机，例如：

| 状态 | 行为 |
| --- | --- |
| `PAIR_TX` | 周期发送配对请求 |
| `PAIR_RX` | 切到接收等待配对响应 |
| `WORK_RX` | 主要时间保持接收遥控命令 |
| `WORK_TX_STATUS` | 短时间切出发送 GPS / 状态包 |
| `WORK_RETURN_RX` | 发完后立刻回接收 |

同时要重新评估：

- GPS / 状态包上报频率是否可以降低。
- 发送状态包期间是否允许短暂收不到遥控包。
- 遥控失联超时阈值是否需要放宽。
- 对端是否支持这种收发时序。

如果对端协议假设船端一直能接收遥控命令，单芯片方案可能需要同步修改对端。

## 8. 需要重点解耦的代码

当前 `wirelessProtocal.c` 不只是协议层，还直接触碰部分业务状态。为了方便移植，建议后续拆成三层：

| 层级 | 建议文件 | 内容 |
| --- | --- | --- |
| 平台层 | `wireless_port.*` | GPIO、SPI、延时、日志、临界区 |
| 芯片层 | `lt8920_driver.*` | 寄存器、FIFO、收发模式 |
| 协议层 | `wireless_protocol.*` | 配对、信道、同步字、帧编解码 |
| 业务层 | `wireless_app.*` | 电机、LED、GPS、自动驾驶接口 |

至少要把下面内容从协议层里隔离出来：

- 电机控制
- LED 控制
- GPS 数据来源
- 电量数据来源
- 自动驾驶状态
- 遥控失联后的业务动作

这样别人换 MCU 或换 RTOS 时，能清楚知道应该改平台层，而不是在协议文件里到处找端口和业务变量。

## 9. 调试建议

### 9.1 先调硬件链路

推荐观察：

- RST 波形是否正确。
- SCK / MOSI / MISO 时序是否正确。
- CS 是否只拉低目标芯片。
- 发送和接收芯片的 MISO 是否读对。
- 电源、晶振、天线匹配是否正常。

### 9.2 再调 LT8920 状态

推荐打印：

- `LT89xx_INIT()` 结果。
- 关键寄存器回读值。
- 当前信道。
- 当前同步字。
- Reg48 的 `PKT_FLAG` 和 `CRC_ERROR`。
- FIFO 返回长度。
- 收到包的十六进制内容。

### 9.3 最后调业务协议

推荐打印：

- 配对阶段发出的 4 字节种子。
- 派生出来的工作信道。
- 派生出来的 key / 同步字。
- 收到的命令字。
- payload 长度。
- 失联计数。

如果现象是“偶尔能配对、长期不稳定”，优先查 SPI 时序、`10ms` 调度周期、电源和天线，不要先怀疑业务解析。

## 10. 移植验收清单

完成移植后至少确认：

- 新 MCU 上不再直接依赖 `STC8G_H_GPIO.h`。
- GPIO / SPI / delay / log 已集中到平台适配层。
- LT8920 寄存器能稳定写入和回读。
- 双芯片方案下两个角色都能独立访问。
- 能稳定完成配对。
- 能切换到工作信道。
- 能稳定接收 `0x11` 控制命令。
- 遥控失联超时逻辑符合预期。
- 能周期发送 `0x12` GPS / 状态包。
- 连续运行不会出现 SPI 偶发读写错误或状态机卡死。

## 11. 当前实现的风险点

这些不是移植时必须立刻修改，但需要提前知道：

1. `Pair_Channel = 0x7F` 按常见 `2402 + channel` MHz 理解会到 `2529MHz`，需要结合 LT8920 手册和项目实测确认。
2. `RF_Encrypt_Config()` 名字不准确，它配置的是 Sync Word，不是真正的数据加密。
3. `RF_key_format()` 依赖固定地址 `0xF4~0xF7` 取种子，换 MCU 后必须定义这 4 字节从哪里来。
4. 当前业务协议文件直接耦合电机、LED、GPS、自动驾驶，不适合作为通用无线库直接复用。
5. 当前代码里存在部分乱码注释，建议移植时统一按 UTF-8 整理。
6. 当前调度依赖 `10ms` tick，换成 RTOS 后不要直接把 tick 数当毫秒。

## 12. 建议补充的资料 

为了让后续换 MCU 的人不用反推代码，建议另外补齐：

- 新旧板子的 LT8920 原理图连接表。
- 双 LT8920 / 单 LT8920 的硬件差异说明。

## 13. 当前 STC32G 工程接入意见

本节针对当前 `Black_Pearl_v1.1` 工程，不是旧 `App/Wireless` 工程的通用描述。结合当前原理图截图，新的无线 / SPI 相关引脚应按下面理解：

| 信号 | 当前引脚 | STC32G 复用功能 | 备注 |
| --- | --- | --- | --- |
| `SPI_CLK` | `P3.2` | `SCLK_4` | 硬件 SPI 第 4 组时钟 |
| `MISO` | `P3.3` | `MISO_4` | 硬件 SPI 第 4 组输入 |
| `MOSI` | `P3.4` | `MOSI_4` | 硬件 SPI 第 4 组输出 |
| `SPI_SS` | `P3.5` | `SS_4` | 片选可由硬件或 GPIO 控制，建议先 GPIO 控制 |
| `RST` | `P5.0` | GPIO | 无线芯片复位脚 |
| `ANT_SEL` | `P5.1` | GPIO | 天线选择脚，按射频前端实际逻辑配置 |
| `RXEN` | `P1.3` | GPIO | KCT8206L 接收通道使能 |
| `TXEN` | `P5.4` | GPIO | KCT8206L 发射通道使能，注意不要开 `MCLKO/SS_3` 复用 |
| `LED` | `P3.6` | GPIO | 当前 `APP_Lamp.c` 也使用 P3.6，注意复用冲突 |

### 13.1 当前工程不是软件 SPI 状态

当前工程里已有 STC32G 硬件 SPI 驱动：

- `Driver/src/STC32G_SPI.c` 使用 `SPCTL`、`SPSTAT`、`SPDAT` 等硬件 SPI 寄存器。
- `Driver/inc/STC32G_SPI.h` 已定义 `SPI_SS_4/P3.5`、`SPI_MOSI_4/P3.4`、`SPI_MISO_4/P3.3`、`SPI_SCLK_4/P3.2`。
- 没有看到面向无线芯片的 GPIO 翻转式软件 SPI 驱动。

但是当前启动配置还没有真正启用这组硬件 SPI：

- `User/System_init.c` 里当前 `SPI_SW(SPI_P22_P23_P24_P25)` 选择的是 P2.2/P2.3/P2.4/P2.5，不匹配当前原理图。
- `User/System_init.c` 里 `SPI_config()` 调用仍被注释，系统启动时没有初始化 SPI 外设。
- `User/Task.c` 里的 `Sample_SPI_PS`、`Sample_DMA_SPI_PS`、`Sample_HSSPI` 都是示例任务且处于注释状态，不属于当前无线业务链路。

因此，当前移植建议是：**按硬件 SPI 方向接入，不建议再新增软件 SPI**。除非后续发现 LT8920 时序必须用特殊 GPIO 控制，才考虑保留软件 SPI 兜底。

### 13.2 建议的最小改动路径

如果当前无线芯片只需要一组 SPI 总线，建议按以下顺序接入：

1. 在 `Switch_config()` 中把 SPI 复用切到第 4 组：`SPI_SW(SPI_P35_P34_P33_P32)`。
2. 在 `SYS_Init()` 中恢复 `SPI_config()` 调用，但先确认主从模式。
3. 如果 MCU 访问无线芯片，`SPI_config()` 应改为 `SPI_Mode_Master`；当前函数里是 `SPI_Mode_Slave`，不适合作为无线芯片主控。
4. 建议 `SPI_SS/P3.5` 先当普通 GPIO 手动控制片选，不依赖硬件自动片选，便于兼容 LT8920 的寄存器 / FIFO 访问时序。
5. 新建无线端口适配层，例如 `wireless_port_stc32g.c/.h`，把 `RST/P5.0`、`ANT_SEL/P5.1`、`CS/P3.5`、SPI 读写、延时和日志统一封装。
6. 先写 LT8920 寄存器读写自检，再接入配对和业务协议，不要一开始直接跑完整无线状态机。

推荐先封装成下面这类接口，避免无线驱动直接散落 `P32/P33/P34/P35/P50/P51`：

| 接口 | 当前 STC32G 实现建议 |
| --- | --- |
| `WIRELESS_CS_WRITE(level)` | 写 `P3.5` |
| `WIRELESS_RST_WRITE(level)` | 写 `P5.0` |
| `WIRELESS_ANT_SEL_WRITE(level)` | 写 `P5.1` |
| `WIRELESS_RXEN_WRITE(level)` | 写 `P1.3` |
| `WIRELESS_TXEN_WRITE(level)` | 写 `P5.4` |
| `WIRELESS_SPI_TRANSFER(byte)` | 调用硬件 SPI 收发，底层访问 `SPDAT` |
| `WIRELESS_DELAY_US(us)` | 调用当前 STC32G delay 接口 |
| `WIRELESS_LOG(...)` | 接当前 `Log` 模块或临时关闭 |

### 13.3 `TXEN/RXEN` 射频前端控制

当前 2.4G 模块不是 LT8920 直接接天线，中间还有 `KCT8206L` 射频前端。`TXEN` 和 `RXEN` 是这个射频前端的收发路径使能脚，不是 SPI，也不是 LT8920 的寄存器信号。

| 信号 | 作用 | 当前 MCU 引脚 | 软件含义 |
| --- | --- | --- | --- |
| `TXEN` | 发射通道使能 | `P5.4` | 发包前拉高，打开发射 PA / TX 路径 |
| `RXEN` | 接收通道使能 | `P1.3` | 收包监听时拉高，打开接收 LNA / RX 路径 |

推荐控制状态：

| `TXEN` | `RXEN` | 状态 | 用途 |
| --- | --- | --- | --- |
| `0` | `0` | 关闭 / 低功耗 | 上电默认、空闲省电 |
| `0` | `1` | 接收模式 | 常态监听遥控包 |
| `1` | `0` | 发射模式 | 短时间发送配对包、状态包 |
| `1` | `1` | 不建议 | 软件应避免同时打开 TX/RX |

推荐软件时序：

1. 无线初始化前先关前端：`TXEN=0, RXEN=0`。
2. 进入监听时：先 `TXEN=0`，再 `RXEN=1`，然后配置 LT8920 进入 RX。
3. 发包前：先 `RXEN=0`，再 `TXEN=1`，延时几个微秒后配置 LT8920 进入 TX 并发送。
4. 发完后：立刻 `TXEN=0`，再 `RXEN=1`，让系统回到接收监听。

如果 `TXEN` 长时间保持高电平，会增加功耗并占用发射链路；如果接收时忘记拉高 `RXEN`，现象通常是 SPI 寄存器正常但空口收不到包。

公开资料中 `KCT8206L` 被描述为集成 PA、LNA、TX/RX switch 和天线分集开关的 2.4G 射频前端；同系列 `KCT8201L` 的控制表也使用 `TX_EN/RX_EN/ANT_SEL` 这类逻辑。因此当前工程应把 `TXEN/RXEN` 作为板级 GPIO 控制，和 LT8920 的 TX/RX 模式切换保持同步。

### 13.4 需要优先确认的冲突点

接入前建议先确认这些点，避免调试时误判为射频问题：

- **SPI 引脚复用冲突**：P3.2/P3.3 也可作为 `I2C_P33_P32`，当前工程 I2C 已切到 `P1.4/P1.5`，不要再把 I2C 切回 P3.2/P3.3。
- **LED 冲突**：当前 `APP_Lamp.c` 描述 LED 在 `P3.6`，如果无线模块 LED 也接 P3.6，需要确认是否仍允许跑 `Sample_Lamp`。
- **RST 脚方向**：`P5.0` 当前在 `GPIO_config()` 中没有显式配置，建议初始化为准双向或推挽输出，并给出上电默认电平。
- **ANT_SEL 脚方向**：`P5.1` 当前也没有显式配置，建议按射频开关要求配置为输出，并明确默认选择内置 / 外置天线哪一路。
- **TXEN/RXEN 方向**：`P5.4`、`P1.3` 需要配置为 GPIO 输出；默认建议都拉低，进入接收后再拉高 `RXEN`。
- **片选策略**：如果只有一颗无线芯片，`P3.5` 可直接作为 CS；如果后续变成双 LT8920，需要重新增加第二个 CS/MISO 或改为单芯片半双工状态机。
- **中断策略**：原文档旧方案有 `PKT` 状态脚概念，当前截图没有看到独立 PKT 引脚，建议第一版先用寄存器轮询收包状态。

### 13.5 当前工程移植优先级

建议按下面优先级推进，而不是直接搬旧 `App/Wireless`：

1. **先打通硬件 SPI**：确认 `P3.2/P3.3/P3.4` 有正确波形，`P3.5` 片选有效。
2. **再打通 LT8920 寄存器**：复位后写一个可回读寄存器，确认 MISO 方向和 SPI 模式正确。
3. **再验证固定包收发**：用固定信道、固定同步字、固定 payload 做空口收发测试。
4. **最后迁移协议层**：只有在 LT8920 驱动稳定后，再接入配对、信道算法、`0x11/0x12` 业务帧。
5. **暂缓业务耦合**：电机、GPS、自动驾驶、LED 等业务先做空接口或日志，避免无线底层未稳定时被业务逻辑干扰。

当前工程已有 QMC6309/QMI8658/GPS/Log 等模块，并且 `SYS_Init()` 里存在 `APP_config()` 后恢复 I2C 的逻辑。无线接入时应尽量把 SPI 和无线 GPIO 初始化放在一个独立函数里，并在 `APP_config()` 之后确认没有被其他模块改写端口模式。
